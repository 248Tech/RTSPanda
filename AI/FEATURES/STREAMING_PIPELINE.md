## Overview

RTSPanda uses mediamtx as an RTSP relay. Go manages mediamtx as a child subprocess.
The browser never connects to cameras directly ‚Äî all streams pass through mediamtx.

```
RTSP Camera
    ‚îÇ
    ‚ñº
mediamtx (child process)
    ‚îÇ   - Ingests RTSP
    ‚îÇ   - Outputs HLS segments to disk (or memory)
    ‚ñº
Go Backend (HTTP server)
    ‚îÇ   - Serves HLS files at /hls/...
    ‚îÇ   - Proxies or redirects to mediamtx HLS port
    ‚ñº
Browser (hls.js player)
```

## mediamtx Configuration

### Template: `mediamtx/mediamtx.yml.tmpl`

Go generates this file at startup and rewrites it when cameras are added/removed.

```yaml
# Generated by RTSPanda ‚Äî do not edit manually

api: yes
apiAddress: 127.0.0.1:9997

hlsAddress: :8888
hlsAlwaysRemux: no
hlsSegmentCount: 3
hlsSegmentDuration: 2s
hlsPartDuration: 200ms

logLevel: warn

paths:
{{- range .Cameras}}
  camera-{{.ID}}:
    source: {{.RTSPURL}}
    sourceOnDemand: yes
    sourceOnDemandCloseAfter: 10s
{{- end}}
```

### Key settings:

| Setting                    | Value       | Why                                                  |
|----------------------------|-------------|------------------------------------------------------|
| `sourceOnDemand`           | yes         | Only connect to camera when someone is watching      |
| `sourceOnDemandCloseAfter` | 10s         | Drop connection 10s after last viewer disconnects    |
| `hlsSegmentCount`          | 3           | Minimal buffer ‚Äî keeps latency low                   |
| `hlsSegmentDuration`       | 2s          | Balance between latency and reliability              |
| `apiAddress`               | 127.0.0.1   | Internal only ‚Äî not exposed outside Go process       |

**Do not change `sourceOnDemand` to `no`** ‚Äî this is a locked decision (see DECISIONS.md ADR-004).

## Go Subprocess Lifecycle

### File: `backend/internal/streams/mediamtx.go`

Responsibilities:
- Generate the mediamtx config file from the camera list
- Start mediamtx as an `exec.Cmd` child process
- Restart mediamtx if it crashes (with backoff)
- Stop mediamtx cleanly on Go shutdown
- Reload config when cameras are added/removed

### Startup sequence:

```
main.go
  ‚îî‚îÄ streams.NewManager(db)
       ‚îú‚îÄ Load cameras from DB
       ‚îú‚îÄ Generate mediamtx config ‚Üí write to DATA_DIR/mediamtx.yml
       ‚îú‚îÄ exec.Command("mediamtx", DATA_DIR/mediamtx.yml)
       ‚îú‚îÄ cmd.Start()
       ‚îî‚îÄ Start watchdog goroutine
```

### Config reload (on camera add/remove/update):

mediamtx supports hot-reload via its API (`POST /v3/config/global/patch` or path-level API).
Use the mediamtx HTTP API to add/remove paths without restarting the process.

Fallback: if the API call fails, write new config file and send SIGHUP to mediamtx.

**Do not restart mediamtx on every camera change** ‚Äî use API patching.

### Watchdog goroutine:

```go
func (m *Manager) watchdog(ctx context.Context) {
    for {
        if err := m.cmd.Wait(); err != nil {
            // process exited
            if ctx.Err() != nil {
                return // intentional shutdown
            }
            log.Printf("mediamtx exited: %v ‚Äî restarting in 3s", err)
            time.Sleep(3 * time.Second)
            m.start()
        }
    }
}
```

## HLS URL Structure

### mediamtx serves HLS at:

```
http://127.0.0.1:8888/camera-{id}/index.m3u8
```

### Go proxies or redirects to the browser at:

```
/hls/camera-{id}/index.m3u8
```

Implementation options (pick one):

**Option A ‚Äî Reverse proxy (recommended):**
Go's `httputil.ReverseProxy` forwards `/hls/` requests to mediamtx port 8888.
Browser sees a single origin. No CORS issues.

**Option B ‚Äî Redirect:**
`GET /hls/...` returns `302` to `http://host:8888/...`.
Simpler, but browser must reach port 8888 directly. Works in local LAN context.

**Default to Option A** (reverse proxy). Keeps everything on port 8080.

## Stream Status API

### File: `backend/internal/streams/health.go`

Poll mediamtx's internal API to determine stream health:

```
GET http://127.0.0.1:9997/v3/paths/list
```

Response includes path state: `idle`, `ready`, `notReady`.

Map to RTSPanda status:

| mediamtx state | RTSPanda status |
|----------------|-----------------|
| `ready`        | `online`        |
| `idle`         | `offline`       |
| `notReady`     | `connecting`    |
| Not in list    | `offline`       |
| API unreachable| `offline`       |

The status endpoint (`GET /api/v1/cameras/:id/stream`) calls this internally.

**Do not poll continuously.** Poll on-demand when the endpoint is called.
Optionally cache the result for 3 seconds to prevent hammering.

## mediamtx Binary

- Download from: https://github.com/bluenviron/mediamtx/releases
- Target: `mediamtx_v1.x.x_linux_amd64.tar.gz` for Docker
- For local dev: download the appropriate platform binary
- Store binary at: `mediamtx/mediamtx` (gitignored)
- In Docker: copy binary into image during build stage

**Do not commit the binary to git.** Add `mediamtx/mediamtx` to `.gitignore`.

The Dockerfile downloads or copies mediamtx in the build stage.

## Acceptance Criteria (TASK-005)

- [ ] mediamtx starts automatically when Go app starts
- [ ] mediamtx stops cleanly when Go app exits (graceful shutdown via context)
- [ ] Watchdog restarts mediamtx if it crashes
- [ ] Adding a camera via API makes its stream available at `/hls/camera-{id}/index.m3u8`
- [ ] Removing a camera removes it from mediamtx config
- [ ] Disabling a camera stops its source connection (sourceOnDemand handles this automatically)
- [ ] Stream status returns `online` / `offline` / `connecting` correctly
